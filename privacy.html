6.4 Automated Database Backups

To protect against data loss from hardware failure, corruption, or other incidents, we maintain a comprehensive automated SQLite backup infrastructure:

6.4.1 Backup Retention Policy

Backups are retained on a tiered schedule to balance recovery granularity with storage efficiency:

  • Hourly snapshots: A database snapshot is taken every hour. The most recent hourly snapshots are available for point-in-time recovery
  • 7 daily backups: One full backup per day is retained for the past 7 days, providing day-level recovery granularity for the previous week
  • 4 weekly backups: One full backup per week is retained for the past 4 weeks, covering the previous month
  • 12 monthly backups: One full backup per month is retained for the past 12 months, enabling recovery from long-term issues

Older backups are automatically rotated and securely deleted when they fall outside the retention window.

6.4.2 Point-in-Time Recovery

The Service supports point-in-time recovery (PITR) using a combination of periodic snapshots and SQLite write-ahead log (WAL) archiving. This allows us to restore the database to any specific moment in time — not just to the last backup — minimising data loss in the event of an incident. PITR is particularly effective for recovering from accidental data corruption or erroneous bulk operations without reverting unrelated changes.

6.4.3 Backup Integrity Verification

Every backup is automatically verified to ensure it can be reliably restored:

  • Checksum validation: A SHA-256 checksum is computed at backup creation and re-verified before any restore operation to detect file corruption or tampering
  • Automated restore tests: Periodic restore drills are performed against an isolated test environment to confirm that backups produce a fully functional, consistent database. These tests validate schema integrity, row counts, and foreign key constraints
  • Alerting: If any backup fails its integrity check, the operations team is notified immediately so corrective action can be taken before the next retention window expires

6.4.4 One-Click Restore from Admin Panel

Authorised administrators can initiate a database restore directly from the admin panel. The restore interface lists all available recovery points (hourly, daily, weekly, monthly) and supports one-click restore with automatic pre-restore backup creation. This ensures that even during a restore operation, no data is irreversibly lost. The admin panel logs all restore operations for audit purposes.

6.4.5 Backup Security

  • All backup files are stored on the same server infrastructure, encrypted at rest, and subject to the same access controls as the primary database
  • Backups are not transferred to or stored on third-party services
  • Access to backup files and restore operations is restricted to authorised administrators and logged for audit

When a user account is deleted, all personal data associated with that account is purged from both the primary database and all existing backup copies within the next scheduled backup cycle (within 1 hour maximum). See Section 7.1 for full details on data erasure.

6.5 WAL Checkpoint Management

The Service uses SQLite's Write-Ahead Logging (WAL) mode for improved concurrent read/write performance. To maintain database health and prevent unbounded WAL file growth, we implement active checkpoint management:

  • Automatic checkpointing: WAL checkpoints are triggered automatically when the WAL file reaches a configured size threshold, transferring committed changes from the WAL back into the main database file
  • Scheduled full checkpoints: A TRUNCATE checkpoint is performed at regular intervals during low-traffic periods to fully reset the WAL file, keeping it compact and ensuring optimal read performance
  • Pre-backup checkpoints: A checkpoint is forced before each backup snapshot to ensure the backup captures a fully consistent state, with no uncommitted WAL data left outside the main database file
  • Graceful shutdown checkpoints: As described in Section 6.6, all WAL entries are flushed and committed during the graceful shutdown sequence before the database connection is closed

This checkpoint strategy ensures that your data is always durably written and that backup files are self-contained and reliable.

6.6 Graceful Shutdown

To safeguard data integrity during server maintenance, updates, or unexpected events, the Service implements a graceful shutdown procedure. When the server process receives a termination signal:

  • Connection draining: All in-flight HTTP requests are allowed to complete before the server stops accepting new connections. No active user sessions are abruptly terminated
  • Database closure: The SQLite database connection is closed cleanly using the better-sqlite3 driver, ensuring all pending write-ahead log (WAL) entries are flushed and committed. This prevents data corruption or partial writes
  • Session persistence: Active session data stored in the SQLite session store is preserved through the shutdown, so users are not forcibly logged out when the server restarts
  • Signal handling: The process listens for SIGTERM and SIGINT signals, triggering the orderly shutdown sequence rather than an abrupt process kill

This means that scheduled maintenance windows, deployments, and server restarts do not result in data loss, corrupted records, or interrupted transactions. Your data remains consistent and intact across all service restarts.

6.7 Data Migration Tooling

To ensure long-term scalability and data portability, the Service includes tooling for migrating data from SQLite to PostgreSQL or other relational database systems if the platform needs to scale beyond SQLite's capabilities:

  • Schema export: The complete database schema (tables, indices, constraints) can be exported and converted to PostgreSQL-compatible DDL statements
  • Data export: All database contents can be exported to standard formats (SQL dump, CSV) for import into a target database system
  • Migration validation: Automated checks verify row counts, data types, and referential integrity after migration to ensure no data is lost or corrupted during the transfer
  • User data portability: Individual user data can be exported independently of a full migration, supporting your Right to Data Portability (Section 8)

In the event that a migration is performed, we will notify all users in advance and ensure that no data is lost during the transition. The Service's privacy protections, backup procedures, and data erasure commitments apply equally regardless of the underlying database technology.

6.4 Cópias de Segurança Automatizadas da Base de Dados

Para proteger contra perda de dados por falha de hardware, corrupção ou outros incidentes, mantemos uma infraestrutura abrangente de cópias de segurança automatizadas SQLite:

6.4.1 Política de Retenção de Cópias de Segurança

As cópias de segurança são retidas num calendário escalonado para equilibrar a granularidade de recuperação com a eficiência de armazenamento:

  • Snapshots horários: Um snapshot da base de dados é realizado a cada hora. Os snapshots horários mais recentes estão disponíveis para recuperação point-in-time
  • 7 cópias diárias: Uma cópia de segurança completa por dia é retida durante os últimos 7 dias, proporcionando granularidade de recuperação ao nível do dia para a semana anterior
  • 4 cópias semanais: Uma cópia de segurança completa por semana é retida durante as últimas 4 semanas, cobrindo o mês anterior
  • 12 cópias mensais: Uma cópia de segurança completa por mês é retida durante os últimos 12 meses, permitindo recuperação de problemas de longo prazo

As cópias de segurança mais antigas são automaticamente rotacionadas e eliminadas de forma segura quando ultrapassam a janela de retenção.

6.4.2 Recuperação Point-in-Time

O Serviço suporta recuperação point-in-time (PITR) utilizando uma combinação de snapshots periódicos e arquivamento do write-ahead log (WAL) do SQLite. Isto permite-nos restaurar a base de dados para qualquer momento específico no tempo — não apenas para a última cópia de segurança — minimizando a perda de dados em caso de incidente. O PITR é particularmente eficaz para recuperar de corrupção acidental de dados ou operações em massa erróneas sem reverter alterações não relacionadas.

6.4.3 Verificação de Integridade das Cópias de Segurança

Cada cópia de segurança é automaticamente verificada para garantir que pode ser restaurada de forma fiável:

  • Validação de checksum: Um checksum SHA-256 é calculado na criação da cópia e re-verificado antes de qualquer operação de restauro para detetar corrupção ou adulteração de ficheiros
  • Testes de restauro automatizados: Exercícios periódicos de restauro são realizados contra um ambiente de teste isolado para confirmar que as cópias produzem uma base de dados totalmente funcional e consistente. Estes testes validam a integridade do esquema, contagens de registos e restrições de chave estrangeira
  • Alertas: Se alguma cópia de segurança falhar a sua verificação de integridade, a equipa de operações é notificada imediatamente para que possam ser tomadas medidas corretivas antes da próxima janela de retenção expirar

6.4.4 Restauro com Um Clique a partir do Painel de Administração

Isto garante que mesmo durante uma operação de restauro, nenhum dado é irreversivelmente perdido. O painel de administração regista todas as operações de restauro para fins de auditoria.

6.5 Gestão de Checkpoints WAL

O Serviço utiliza o modo Write-Ahead Logging (WAL) do SQLite para melhorar o desempenho de leitura/escrita concorrente. Para manter a saúde da base de dados e prevenir o crescimento ilimitado do ficheiro WAL, implementamos gestão ativa de checkpoints:

  • Checkpointing automático: Os checkpoints WAL são acionados automaticamente quando o ficheiro WAL atinge um limiar de tamanho configurado, transferindo as alterações confirmadas do WAL de volta para o ficheiro principal da base de dados
  • Checkpoints completos agendados: Um checkpoint TRUNCATE é realizado em intervalos regulares durante períodos de baixo tráfego para resetar completamente o ficheiro WAL, mantendo-o compacto e garantindo desempenho de leitura ótimo
  • Checkpoints pré-cópia de segurança: Um checkpoint é forçado antes de cada snapshot de cópia de segurança para garantir que a cópia captura um estado totalmente consistente, sem dados WAL não confirmados fora do ficheiro principal da base de dados
  • Checkpoints de encerramento gracioso: Conforme descrito na Secção 6.6, todas as entradas WAL são descarregadas e confirmadas durante a sequência de encerramento gracioso antes de a ligação à base de dados ser encerrada

Esta estratégia de checkpoints garante que os seus dados são sempre escritos de forma durável e que os ficheiros de cópia de segurança são autónomos e fiáveis.

6.6 Encerramento Gracioso

Para salvaguardar a integridade dos dados durante manutenção do servidor, atualizações ou eventos inesperados, o Serviço implementa um procedimento de encerramento gracioso. Quando o processo do servidor recebe um sinal de terminação:

  • Drenagem de ligações: Todos os pedidos HTTP em curso são autorizados a completar antes de o servidor parar de aceitar novas ligações. Nenhuma sessão de utilizador ativa é terminada abruptamente
  • Fecho da base de dados: A ligação à base de dados SQLite é encerrada de forma limpa utilizando o driver better-sqlite3, garantindo que todas as entradas pendentes do write-ahead log (WAL) são descarregadas e confirmadas. Isto previne corrupção de dados ou escritas parciais
  • Persistência de sessões: Os dados de sessão ativos armazenados no armazenamento de sessões SQLite são preservados durante o encerramento, para que os utilizadores não sejam forçados a terminar sessão quando o servidor reinicia
  • Tratamento de sinais: O processo escuta sinais SIGTERM e SIGINT, acionando a sequência de encerramento ordenado em vez de uma terminação abrupta do processo

Isto significa que janelas de manutenção agendadas, implementações e reinícios do servidor não resultam em perda de dados, registos corrompidos ou transações interrompidas. Os seus dados permanecem consistentes e intactos durante todos os reinícios do serviço.

6.7 Ferramentas de Migração de Dados

Para garantir escalabilidade a longo prazo e portabilidade dos dados, o Serviço inclui ferramentas para migrar dados do SQLite para PostgreSQL ou outros sistemas de bases de dados relacionais, caso a plataforma necessite de escalar para além das capacidades do SQLite:

  • Exportação de esquema: O esquema completo da base de dados (tabelas, índices, restrições) pode ser exportado e convertido para instruções DDL compatíveis com PostgreSQL
  • Exportação de dados: Todo o conteúdo da base de dados pode ser exportado para formatos padrão (dump SQL, CSV) para importação num sistema de base de dados de destino
  • Validação de migração: Verificações automatizadas confirmam contagens de registos, tipos de dados e integridade referencial após a migração para garantir que nenhum dado é perdido ou corrompido durante a transferência
  • Portabilidade de dados do utilizador: Os dados individuais do utilizador podem ser exportados independentemente de uma migração completa, apoiando o seu Direito à Portabilidade de Dados (Secção 8)

No caso de uma migração ser realizada, notificaremos todos os utilizadores com antecedência e garantiremos que nenhum dado é perdido durante a transição. As proteções de privacidade, procedimentos de cópia de segurança e compromissos de eliminação de dados do Serviço aplicam-se igualmente independentemente da tecnologia de base de dados subjacente.

7. Retenção de Dados

Retemos os seus dados pelos seguintes períodos:

  • Dados da conta: Retidos enquanto a sua conta estiver ativa
  • Dados de análise: Retidos de forma agregada e não identificável. Os dados de sessão individuais são periodicamente eliminados
  • Tokens de verificação de email e redefinição de palavra-passe: Expiram automaticamente e são eliminados após o seu período de validade
  • Dados de sessão: As sessões expiradas são periodicamente limpas da base de dados
  • Dados de alertas de preço: Retidos enquanto a sua conta estiver ativa e os alertas estiverem configurados

7.1 Direito ao Esquecimento — Eliminação Completa de Dados na Eliminação da Conta

Quando elimina a sua conta, realizamos a eliminação completa e permanente de todos os dados pessoais associados à sua conta. Não se trata de uma desativação ou eliminação temporária — os seus dados são irreversivelmente removidos dos nossos sistemas. Especificamente, os seguintes dados são permanentemente eliminados:

  • Informações da conta: O seu nome, endereço de email, plano de subscrição e todos os metadados da conta (carimbos de data/hora do último login, contagens de login, tentativas de login falhadas, registos de bloqueio)
  • Dados de autenticação: Hashes de palavras-passe, ligações e identificadores de fornecedores OAuth, segredos TOTP de autenticação de dois fatores
  • Dados de perfil: Imagens de avatar e seleções de cor de fundo do avatar
  • Definições e preferências do utilizador: Preferências de alertas de preço, estado de subscrição da newsletter semanal, preferência de modo escuro, preferências de notificação (email, na aplicação, push)
  • Regras de alertas de preço: Todas as regras de alertas de preço configuradas e as suas condições associadas
  • Notificações: Todas as notificações na aplicação enviadas para a sua conta
  • Sessões ativas: Todos os registos de sessão do lado do servidor ligados à sua conta
  • Tokens de segurança: Quaisquer tokens pendentes de verificação de email e redefinição de palavra-passe
O nosso compromisso: Não retemos perfis sombra ou dados residuais após a eliminação da conta. Os seus dados pessoais são purgados da base de dados principal imediatamente e de todas as cópias de segurança no próximo ciclo de cópia de segurança horária (máximo 1 hora). Uma vez totalmente purgados, não existe forma de recuperar os seus dados. Esta eliminação é irreversível.

Dados de análise anonimizados e agregados (visualizações de página, eventos) que não podem ser associados à sua identidade podem ser retidos para fins de melhoria do serviço, uma vez que estes dados não constituem dados pessoais.

8. Os Seus Direitos

Tem os seguintes direitos relativamente aos seus dados pessoais:

  • Direito de acesso: Pode solicitar uma cópia dos dados pessoais que detemos sobre si
  • Direito de retificação: Pode solicitar que corrijamos dados pessoais imprecisos ou incompletos. Também pode atualizar o seu nome, email, avatar e preferências diretamente através das definições da conta
  • Direito ao apagamento (Direito ao Esquecimento): Tem o direito de solicitar a eliminação completa e permanente de todos os dados pessoais que detemos sobre si. Quando elimina a sua conta — seja através das definições da conta ou contactando-nos — todos os dados listados na Secção 7.1 são permanente e irreversivelmente apagados dos nossos sistemas, incluindo todas as cópias de segurança (purgadas no próximo ciclo de cópia de segurança horária). Nenhum dado residual ou perfil sombra é retido. Pode também solicitar o apagamento sem eliminar a sua conta contactando-nos diretamente, caso em que eliminaremos todos os dados que não sejam estritamente necessários para o funcionamento da conta
  • Direito à portabilidade de dados: Pode solicitar uma exportação legível por máquina dos seus dados pessoais antes da eliminação da conta
  • Direito de retirar o consentimento: Quando o tratamento se baseia no consentimento (ex.: cookies de análise, newsletter), pode retirar o consentimento a qualquer momento sem afetar a legalidade do tratamento anterior
  • Direito de oposição: Pode opor-se ao tratamento dos seus dados pessoais em determinadas circunstâncias
  • Direito à limitação do tratamento: Pode solicitar que limitemos o tratamento dos seus dados pessoais enquanto uma reclamação ou pedido estiver a ser resolvido

Para exercer qualquer destes direitos, contacte-nos usando os detalhes fornecidos na Secção 12 abaixo. Responderemos ao seu pedido dentro de 30 dias. Os pedidos de eliminação de conta e apagamento completo de dados são processados imediatamente após confirmação.

Não utilizamos quaisquer serviços de análise de terceiros. Todas as análises são processadas pelo nosso sistema auto-hospedado, e nenhum dado de análise é partilhado com partes externas.
