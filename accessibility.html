Our commitment

OIL Benchmarks is committed to ensuring digital accessibility for people with disabilities. We continually improve the user experience for everyone and apply the relevant accessibility standards to make our content and functionality available to the widest possible audience.

This statement applies to the web application at oilbenchmarks.com, including all pages served by the application and the installable Progressive Web App (PWA).

Conformance status

We target conformance with the Web Content Accessibility Guidelines (WCAG) 2.2 at Level AA. This is also the technical standard referenced by the European Accessibility Act (EAA), Directive (EU) 2019/882, enforceable from 28 June 2025, and by EN 301 549 v3.2.1, the harmonised European standard for ICT accessibility.

Current self-assessed status: Partially conformant

"Partially conformant" means that some portions of the content do not yet fully conform to the standard. Known gaps are listed below.

Accessibility features

The following features are implemented across the application:

Perceivable

  • Text alternatives: all informational images, icons, and chart elements include alt text or aria-label descriptions.
  • Colour contrast: the default dark theme uses gold (#c9a84c) on dark (#0c0c0e) with a contrast ratio of 7.4:1 for body text, exceeding the AA minimum of 4.5:1.
  • Resizable text: all text sizes are defined in relative units. The layout remains usable at 200% zoom.
  • Colour independence: status indicators (on/off, up/down) use text labels and icons alongside colour, so information is not conveyed by colour alone.

Operable

  • Keyboard navigation: all interactive elements (buttons, links, form fields, dialogs, chart controls) are reachable and operable via keyboard alone.
  • Focus indicators: visible focus rings are applied to all focusable elements, with a minimum 2 px outline offset.
  • Skip links: a "Skip to main content" link is provided as the first focusable element.
  • No keyboard traps: modal dialogs (delete confirmation, restore confirmation) trap focus while open and return focus to the trigger element on close.
  • Timeouts: sessions have configurable timeouts; users receive a warning notification before session expiry when push notifications are enabled.

Understandable

  • Language: the page language is declared (lang="en") and the admin panel supports English/Portuguese switching.
  • Consistent navigation: the bottom navigation bar (mobile) and sidebar (desktop) follow the same order on every page.
  • Error identification: form validation errors are announced with descriptive messages adjacent to the field, associated via aria-describedby.
  • Labels: all form inputs have visible labels or accessible names via <label> or aria-label.

Robust

  • Semantic HTML: pages use landmarks (<header>, <main>, <nav>, <section>), headings in logical order, and ARIA roles where native semantics are insufficient.
  • ARIA patterns: toggle switches, tab panels, toast notifications (role="status"), and alert dialogs follow WAI-ARIA Authoring Practices.
  • Progressive enhancement: the offline fallback and service worker are designed to degrade gracefully, with context injected via window.__OFFLINE_CONTEXT for screen readers.

Known limitations

We are aware of the following gaps and are actively working to address them:

Area Issue WCAG criteria Status
Price chart Canvas-rendered chart data is not fully exposed to assistive technology. A data table alternative is provided below the chart, but live tooltip values are not announced. 1.1.1, 4.1.2 In progress
Admin panel charts Chart.js canvases in the admin dashboard lack structured data table equivalents for some chart types. 1.1.1 Planned
PDF export Admin PDF reports are not tagged for accessibility. Screen readers may not correctly parse the document structure. 1.3.1, 4.1.2 Planned
CAPTCHA Image CAPTCHA on registration is not accessible to users with visual impairments. An audio alternative is not yet available. 1.1.1, 1.3.1 Evaluating alternatives
Touch targets Some compact table action buttons in the admin panel are below the recommended 44 × 44 CSS pixel minimum on mobile viewports. 2.5.8 In progress

Assessment methodology

We evaluate accessibility using a combination of:

  • Automated scanning: axe-core (integrated into CI), Lighthouse Accessibility audit, WAVE browser extension.
  • Manual testing: keyboard-only navigation, screen reader testing with VoiceOver (macOS/iOS) and NVDA (Windows), high-contrast mode, 200% text zoom, reduced-motion preference.
  • Code review: ARIA usage and semantic HTML are checked during pull request review.

Compatibility

The application is designed to be compatible with:

  • Recent versions of Chrome, Firefox, Safari, and Edge.
  • VoiceOver on macOS and iOS Safari.
  • NVDA and JAWS on Windows with Chrome and Firefox.
  • TalkBack on Android Chrome.
  • OS-level high-contrast / forced-colours modes.
  • prefers-reduced-motion and prefers-color-scheme media queries.

Legal framework

This statement is provided in accordance with:

  • European Accessibility Act (EAA) — Directive (EU) 2019/882, applicable from 28 June 2025 to products and services offered to consumers in the EU, including websites and mobile applications.
  • EN 301 549 v3.2.1 — harmonised European standard for accessibility requirements of ICT products and services, which incorporates WCAG 2.1 Level AA and extends it with additional requirements for software and non-web documents.
  • Web Accessibility Directive (EU) 2016/2102 — applicable to public-sector bodies; included here as a reference standard.

While OIL Benchmarks is not a public-sector body, we voluntarily align with these standards to serve all users equitably and to comply with the EAA's requirements for digital services.

Feedback

Email: [email protected]

Response time: We aim to respond within 5 business days and to provide an accessible alternative within 15 business days.

Escalation: If you are not satisfied with our response, you may contact the relevant national enforcement body in your EU member state. A list of enforcement bodies is maintained by the European Commission.

Continuous improvement

We review this statement and conduct a fresh accessibility audit at least every six months. Improvements are tracked in our issue tracker and prioritised alongside security and performance work.

Our goal is to reach full WCAG 2.2 Level AA conformance and to resolve all known limitations listed above.
